# 05 — Non-Functional Requirements

## 1. Performance

| Metric | Requirement |
|--------|-------------|
| API response time (p95) | < 300ms for standard queries |
| Search results | < 500ms including availability check |
| Page load (First Contentful Paint) | < 1.5s on 3G connection |
| Database query time (p99) | < 100ms with proper indexing |
| File upload (images) | < 5s for images up to 5MB |
| Concurrent users | Support 10,000 concurrent users |

### Performance Strategies
- Database connection pooling (SQLAlchemy `pool_size=20`)
- Redis caching for hotel search results (TTL: 5 minutes)
- Next.js ISR (Incremental Static Regeneration) for hotel pages
- CDN for static assets and images (Cloudflare)
- Async/await throughout FastAPI for non-blocking I/O
- Pagination on all list endpoints (default page_size=20)

## 2. Scalability

## 4. Security

### Authentication & Authorization
- JWT tokens with short TTL (access: 15 min, refresh: 7 days)
- Refresh tokens stored as httpOnly cookies (XSS protection)
- CSRF protection on state-changing requests
- Role-based access control enforced at API layer

### Data Security
- All passwords hashed with bcrypt (cost factor 12)
- Database credentials in environment variables (never hardcoded)
- Sensitive config in secrets manager (AWS Secrets Manager / Vault)
- HTTPS enforced everywhere (HSTS header)
- SQL injection prevention via SQLAlchemy parameterized queries
- Input validation via Pydantic on all API inputs

### Tenant Isolation
- All DB queries scoped with `tenant_id`
- API endpoints validate that requested resources belong to the caller's tenant
- Super Admin endpoints require separate authentication scope

### Rate Limiting
| Endpoint Category | Limit |
|-------------------|-------|
| Auth (login/register) | 5 requests/minute per IP |
| Search API | 60 requests/minute per user |
| Booking API | 10 requests/minute per user |
| General API | 120 requests/minute per user |

### Audit Logging
- All admin actions logged with: user_id, tenant_id, action, timestamp, IP
- Super Admin actions logged separately
- Logs retained for 90 days

## 5. Usability

- Mobile-first responsive design
- WCAG 2.1 Level AA accessibility compliance
- Support for latest 2 versions of major browsers (Chrome, Firefox, Safari, Edge)
- Multi-language ready (i18n framework in place, English as MVP)
- Keyboard navigable UI
- Error messages must be human-readable and actionable

## 6. Maintainability

- Code coverage target: 80% for backend unit tests
- All APIs documented via OpenAPI/Swagger (auto-generated by FastAPI)
- Semantic versioning for API (`/api/v1/...`)
- Environment-based configuration (`.env` files)
- Docker-based local development
- CI/CD pipeline (GitHub Actions)

### Code Quality Standards
- Python: Black formatter, isort, flake8, mypy type hints
- TypeScript: ESLint + Prettier
- Pre-commit hooks enforcing standards
- PR reviews required before merge to main

---

## 7. Data Integrity

- Foreign key constraints enforced at DB level
- Soft delete pattern (`is_deleted = true`) for critical records
- Optimistic locking for booking conflicts (check availability again at commit)
- Database transactions for multi-step operations (booking + payment)
- Input validation at API layer (Pydantic) AND database layer (constraints)

## 9. Observability

### Monitoring Stack
- **Metrics**: Prometheus + Grafana
- **Logging**: Structured JSON logs → ELK Stack (Elasticsearch, Logstash, Kibana)
- **Tracing**: OpenTelemetry → Jaeger
- **Alerting**: PagerDuty / Slack alerts for p95 > 500ms, error rate > 1%

### Key Metrics to Track
- Request rate, error rate, latency (RED method)
- Booking conversion funnel
- Tenant usage (active bookings, revenue)
- DB connection pool utilization
- Background job queue depth

## 10. Disaster Recovery

```
Backup Strategy:
├── MySQL: Daily full backup + binary log replication
├── Media (S3): Cross-region replication enabled
├── Application config: Git-versioned
└── Secrets: Replicated in secondary region

Recovery Playbook:
1. Detect failure (automated monitoring alert)
2. Assess impact scope
3. Switch DNS to standby region (< 5 min with Route53)
4. Restore DB from latest backup if needed
5. Verify application health
6. Notify affected tenants
```
